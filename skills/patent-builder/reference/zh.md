# Patent Builder（中文版）

用于帮助用户发现、打磨、评估并整理潜在专利方案。

支持三种启动方式：
1. 已有工程实现
2. 已有初步想法
3. 没有明确方向，需要启发式引导

不作法律结论。所有专利性判断均为初步技术分析；涉及申请、权利要求布局、侵权风险、自由实施或法律意见时，应建议咨询专业专利代理人或律师。

## 核心原则

- 聚焦技术问题、技术手段和技术效果。
- 区分"已确认实现"和"可选实施方式"。
- 不夸大新颖性、创造性或授权可能性。
- 除非用户提供，不使用真实项目、真实产品、真实代码或真实案例。
- 使用中性占位表达，例如"所述系统""处理模块""目标文档""数据对象""候选对象""步骤 N""对象 A"。
- 区分常规技术手段与可能具有创造性的技术组合。
- 优先描述具体机制，避免仅写"使用 AI""提升体验"等抽象表述。
- 输出应结构化、可复用、便于转化为交底书。

## 启动方式判断

先判断用户从哪种状态开始。

**模式 A：已有工程实现**
适用于用户提供了系统、功能、原型、架构、代码描述、技术文档、产品流程或实现细节。

**模式 B：已有初步想法**
适用于用户已有初步概念、痛点或可能方案，但尚无实现。

**模式 C：没有明确方向**
适用于用户没有清晰 idea，希望通过提问启发产生候选发明方向。

如果不明确，只问一个简短问题：
"我们是从已有工程实现、初步想法，还是开放式探索开始？"

## 通用工作流

根据用户材料深度灵活使用以下清单。

```text
Patent Builder Progress:
- [ ] 1. 明确技术领域和目标场景
- [ ] 2. 澄清技术问题
- [ ] 3. 识别已知或常规方案
- [ ] 4. 提取区别性技术特征
- [ ] 5. 构建候选发明概念
- [ ] 6. 初步评估新颖性、创造性、可实施性和价值
- [ ] 7. 选择最强发明方向
- [ ] 8. 起草或优化专利交底材料
- [ ] 9. 列出待确认问题
```

## 模式 A：已有工程实现

### 目标
从已有系统、功能、产品流程、代码库、架构或设计中提炼可专利的技术组合。

### 流程

1. **拆解工程实现**
   识别：
   - 输入与输出
   - 核心模块
   - 数据结构
   - 处理流程
   - 决策逻辑
   - 存储或索引逻辑
   - 模型、算法或规则的参与方式
   - 前端、后端或用户交互链路

2. **抽象技术问题**
   将实现细节转化为专利式技术问题：
   - 解决了何种失败、低效、成本、错配、延迟、不准确、不稳定或不一致？
   - 该问题受什么技术约束影响？
   - 系统行为获得了什么改进？

3. **区分常规实现与创造性组合**
   将特征标记为：
   - 可能属于常规技术
   - 属于具体实现但不一定有创造性
   - 单独可能有创造性
   - 作为组合可能有创造性

4. **寻找候选创造性组合**
   优先寻找闭环机制，例如：
   - 检测 → 评分 → 排序 → 选择 → 生成 → 校验
   - 提取 → 分类 → 关联 → 检索 → 约束 → 输出
   - 规划 → 执行 → 验证 → 回退
   - 索引 → 检索 → 重排 → 决策 → 展示

5. **起草交底书**
   包括：
   - 背景
   - 发明要点
   - 具体描述
   - 技术效果
   - 可选附图
   - 待确认问题

### 模式 A 输出
提供：
- 工程实现摘要
- 技术问题抽象
- 候选创造性特征
- 最强发明方向
- 已知方案对比
- 交底书草稿或优化建议

## 模式 B：已有初步想法

### 目标
将初步 idea 转化为可实施技术方案和候选专利交底书。

### 流程

1. **澄清 idea**
   询问或推断：
   - 要解决什么问题？
   - 影响谁或什么对象？
   - 现在通常怎么做？
   - 当前做法哪里不理想？
   - 系统最终应输出什么结果？

2. **结构化 idea**
   转化为：
   - 输入数据
   - 处理模块
   - 中间结果
   - 决策规则或评分逻辑
   - 最终输出
   - 反馈、校验或回退逻辑

3. **补足可实施细节**
   在缺失处补充：
   - 数据表示
   - 匹配或评分方法
   - 排序或选择标准
   - 阈值或约束
   - 状态管理
   - 验证和恢复行为

4. **生成发明变体**
   形成 2-3 个候选方向：
   - 算法或流程方向
   - 系统架构方向
   - 交互或控制闭环方向

5. **评估并选择**
   从以下维度评分：
   - 技术具体性
   - 与常规方案的差异
   - 可实施性
   - 技术效果
   - 撰写强度

### 模式 B 输出
提供：
- 精炼后的问题陈述
- 可实施技术方案
- 候选发明方向
- 推荐方向
- 交底书草稿
- 待确认问题

## 模式 C：开放式探索

### 目标
从没有明确方向，引导用户产生一个或多个可继续发展的候选专利 idea。

### 流程

1. **选择领域或上下文**
   引导用户确定一个技术领域、产品区域、工作流程、用户群体或技术环境。

2. **发现痛点**
   探索：
   - 重复性工作
   - 易错决策
   - 高延迟或高成本
   - 低准确率
   - 数据利用不足
   - 依赖人工判断
   - 流程割裂
   - 资源或约束受限

3. **将痛点转化为发明种子**
   使用发明模式：
   - 多源证据融合
   - 先结构化规划再生成或执行
   - 动态评分与预算选择
   - 用轻量引用替代重数据传输
   - 基于置信度的回退
   - 上下文感知路由
   - 用户状态或设备状态自适应
   - 可验证中间计划

4. **创建候选 idea**
   每个候选包含：
   - 问题
   - 技术机制
   - 所需数据
   - 输出
   - 与常规方案的区别
   - 技术效果
   - 待确认问题

5. **选择一个方向继续发展**
   推荐最强 idea，并继续使用模式 B。

### 模式 C 输出
提供：
- 痛点清单
- 候选发明 idea
- 专利潜力排序
- 推荐方向
- 继续发展该 idea 所需的问题

## 交底书结构

除非用户提供其它模板，否则使用以下结构。

### 1. 背景
包括：
- 技术领域
- 要解决的问题
- 已知或常规方案
- 已知方案缺点
- 为什么需要额外方案

### 2. 发明要点
包括：
- 核心技术思路
- 主要模块或步骤
- 区别性特征
- 相对已有方案的优势
- 技术效果

### 3. 具体描述
包括：
- 系统架构
- 处理流程
- 数据结构
- 评分、匹配、排序或决策逻辑
- 可选实施方式
- 回退或校验逻辑
- 使用中性占位符的实施例

### 4. 可选附图
可建议：
- 总体架构图
- 数据流图
- 构建或预处理阶段流程
- 查询或响应阶段流程
- 评分或决策流程
- 输出或展示流程

### 5. 待确认问题
列出最终定稿前需要确认的事实：
- 哪些已经实现？
- 哪些是可选方案？
- 哪些数据可获得？
- 哪些部分是规则、模型或混合实现？
- 哪个是主流程，哪个是兜底流程？
- 哪些点应宽保护？
- 哪些点应作为实施例？

## 输出模板

### 技术问题模板
```text
技术问题在于：[系统/流程] 在 [技术约束] 下无法可靠实现 [期望技术结果]，导致 [技术失败或低效]。
```

### 发明概念模板
```text
所述方案通过以下方式执行 [技术过程]：
1. [技术步骤 1]
2. [技术步骤 2]
3. [技术步骤 3]

相较于 [已知方案]，该方案产生 [技术效果]。
```

### 已知方案对比模板
```text
已知方案：[常规机制]
不足：[技术缺陷]
本方案区别：[区别性技术特征]
技术效果：[技术改进]
```

### 交底书摘要模板
```text
本发明涉及 [技术领域]。其通过 [核心技术机制] 解决 [技术问题]。系统包括 [主要组成]。相较于常规方案，本发明能够改善 [技术效果]。
```

## 质量检查

在完成任何专利导向输出前，确认：

- 技术问题具体明确。
- 方案包含具体技术步骤。
- 与常规方案的差异明确。
- 没有仅以商业规则或抽象 idea 表述发明。
- 已确认实现与可选实施方式已分开。
- 未使用用户未提供的真实案例。
- 避免无依据使用"全新""独特""必然可授权"等表述。
- 法律或申请相关建议均提示需要专业人士确认。
- 已列出待确认问题。

## 不确定性处理

事实未确认时使用以下标签：

- **已确认：** 用户事实支持。
- **假设：** 基于上下文推断但未确认。
- **可选实施方式：** 可能的实现变体。
- **需确认：** 申请或最终撰写前必须核实。

## 常见误区

- 不要把通用 AI 应用直接当作发明，除非有具体技术机制。
- 不要将占位符、索引、缓存、向量检索或分类器单独视为必然有创造性。
- 不要用"AI 处理数据"替代具体流程。
- 不要忽略实现约束。
- 不要把保护范围过度限定为单一实施例。
- 除非用户明确提供并要求使用，不要包含机密项目名。
- 除非用户提供或明确要求，不要使用真实案例。

## 推荐最终输出格式

### 分析类任务
```text
1. 技术问题
2. 已有或常规方案
3. 候选创造性特征
4. 最强发明概念
5. 技术效果
6. 风险与待确认问题
7. 建议下一步
```

### 交底书撰写类任务
```text
1. 背景
2. 发明要点
3. 具体描述
4. 可选附图
5. 待确认问题
```

### 头脑风暴类任务
```text
1. 澄清问题
2. 痛点
3. 候选 idea
4. 专利潜力排序
5. 推荐继续发展的 idea
```
